## 项目结构
数据库存储后端位于session/database目录，采用接口隔离与分层设计：
- 接口层：session包定义Session与SessionService接口，统一抽象不同后端行为。
- 数据库实现层：database包提供基于SQLite的实现，使用sqlx进行SQL操作。

- 测试与工具：包含单元测试、Schema初始化脚本与Snowflake ID生成器。

## 架构总览
数据库后端采用"服务+会话"两级结构：
- 服务层：DatabaseSessionService负责会话生命周期管理（创建/删除/获取），内部通过sqlx.DB执行SQL。
- 会话层：databaseSession封装具体会话的数据访问，提供消息的插入、删除、分页查询、全量查询、归档查询与归档事务。

### databaseSession 数据访问与事务
- 表与列：
  - sessions：主键session_id，记录创建/更新/删除时间戳。
  - messages：主键message_id，包含角色、名称、内容、推理内容、工具调用JSON、工具调用ID、Token用量、创建/更新/归档/删除时间戳。
- 查询与操作：
  - 创建消息：INSERT，所有活跃消息（未归档且未删除）的compacted_at=0。
  - 删除消息：DELETE，软删除（deleted_at=0时才允许删除）。
  - 分页查询：按created_at升序，限制条数与偏移。
  - 全量查询：同上但无LIMIT/OFFSET。
  - 归档查询：仅返回compacted_at>0且未删除的消息。
  - 归档事务：BEGIN TX -> 更新所有活跃消息的compacted_at -> 插入摘要消息 -> COMMIT。
- 并发控制：通过数据库事务保证归档过程原子性；未见显式锁或行级锁，依赖数据库事务隔离级别。

## SQLite数据库集成

### 数据库连接管理
数据库后端使用sqlx库进行SQLite数据库操作，支持以下特性：
- **连接池管理**：通过sqlx.DB提供连接池功能，支持最大连接数、空闲连接数和连接生命周期配置。
- **上下文支持**：所有数据库操作都支持context.Context，便于超时控制和取消操作。
- **事务处理**：使用sql.TxOptions进行事务管理，支持BEGIN/COMMIT/ROLLBACK操作。

### SQLite驱动配置
- **驱动选择**：使用github.com/mattn/go-sqlite3驱动，支持完整的SQLite功能集。
- **内存数据库**：测试环境中使用`:memory:`作为数据库URL，实现完全内存化的测试环境。
- **文件数据库**：生产环境中使用文件路径作为数据库URL，实现持久化存储。

### 初始化流程
数据库初始化通过以下步骤完成：
1. 连接SQLite数据库
2. 创建sessions表
3. 创建messages表
4. 设置必要的约束和默认值

## 消息持久化机制

### ToolCalls JSON序列化
消息持久化采用自定义的JSON序列化机制：
- **ToolCalls类型**：定义为ToolCall结构体切片
- **JSON序列化**：通过实现driver.Valuer接口将切片序列化为JSON字符串
- **JSON反序列化**：通过实现sql.Scanner接口从JSON字符串反序列化
- **存储格式**：tool_calls字段存储为TEXT类型的JSON数组，默认值为'[]'

### Message结构设计
Message结构体包含完整的对话消息信息：
- **基础字段**：MessageID、Role、Name、Content
- **推理字段**：ReasoningContent用于存储链式思维输出
- **工具调用字段**：ToolCalls、ToolCallID支持工具调用的关联
- **Token统计**：PromptTokens、CompletionTokens、TotalTokens记录使用量
- **时间戳字段**：CreatedAt、UpdatedAt、CompactedAt、DeletedAt跟踪状态

### 序列化实现细节

```go
// ToolCalls实现driver.Valuer接口
func (tl ToolCalls) Value() (driver.Value, error) {
    b, err := json.Marshal(tl)
    if err != nil {
        return nil, err
    }
    return string(b), nil
}

// ToolCalls实现sql.Scanner接口
func (tl *ToolCalls) Scan(src any) error {
    if src == nil {
        *tl = ToolCalls{}
        return nil
    }
    var s string
    switch v := src.(type) {
    case string:
        s = v
    case []byte:
        s = string(v)
    default:
        return fmt.Errorf("message: unsupported ToolCallList source type: %T", src)
    }
    return json.Unmarshal([]byte(s), tl)
}
```

## 数据库Schema设计

### Sessions表结构
sessions表用于存储会话元数据：
- **session_id** (INTEGER PRIMARY KEY)：会话唯一标识符
- **created_at** (INTEGER NOT NULL)：会话创建时间戳
- **updated_at** (INTEGER NOT NULL)：会话最后更新时间戳
- **deleted_at** (INTEGER NOT NULL)：会话删除时间戳，0表示未删除

### Messages表结构
messages表用于存储对话消息：
- **message_id** (INTEGER PRIMARY KEY)：消息唯一标识符
- **role** (TEXT NOT NULL DEFAULT '')：消息角色（system/user/assistant/tool）
- **name** (TEXT NOT NULL DEFAULT '')：消息发送者名称
- **content** (TEXT NOT NULL DEFAULT '')：消息内容
- **reasoning_content** (TEXT NOT NULL DEFAULT '')：推理内容
- **tool_calls** (TEXT NOT NULL DEFAULT '[]')：工具调用JSON
- **tool_call_id** (TEXT NOT NULL DEFAULT '')：关联的工具调用ID
- **prompt_tokens** (INTEGER NOT NULL DEFAULT 0)：提示Token数量
- **completion_tokens** (INTEGER NOT NULL DEFAULT 0)：生成Token数量
- **total_tokens** (INTEGER NOT NULL DEFAULT 0)：总Token数量
- **created_at** (INTEGER NOT NULL)：消息创建时间戳
- **updated_at** (INTEGER NOT NULL)：消息最后更新时间戳
- **compacted_at** (INTEGER NOT NULL DEFAULT 0)：归档时间戳
- **deleted_at** (INTEGER NOT NULL)：删除时间戳

## 性能考量
- **读写速度**：
  - 内存后端：O(1)到O(n)的数组操作，适合小规模历史；分页查询为切片截取。
  - 数据库后端：基于SQLite的磁盘IO，分页查询通过LIMIT/OFFSET实现；活跃消息查询按created_at排序。
- **存储容量**：
  - 内存后端受进程内存限制；数据库后端可扩展至磁盘空间。
- **可靠性**：
  - 数据库后端支持事务与软删除，归档过程原子性保障较好。
- **并发控制**：
  - 未见显式锁；归档通过事务保证一致性；建议在应用层避免并发修改同一会话的历史。
- **查询优化建议**：
  - 为sessions(session_id)与messages(message_id)建立索引。
  - 为messages(deleted_at, compacted_at, created_at)复合索引以优化活跃消息查询。
  - 对频繁过滤条件建立索引，如deleted_at、compacted_at组合索引。
  - 合理设置SQLite WAL模式与同步策略以提升并发与可靠性。
- **批量操作**：
  - 归档使用单事务批量更新与插入，减少多次往返开销。
- **连接池与监控**：
  - 建议使用sqlx.DB连接池参数（最大打开连接、最大空闲连接、连接生命周期）。
  - 结合应用监控埋点统计慢查询与事务耗时。

## 故障排查指南
- **会话不存在**：
  - GetSession返回nil表示未找到，属于正常流程，上层应据此创建新会话。
- **删除失败**：
  - DeleteMessage要求deleted_at=0，若消息已被软删除则无法再次删除。
- **归档异常**：
  - CompactMessages在事务中执行，任一步骤失败会回滚；检查compactor输出是否合法。
- **测试验证**：
  - 单测覆盖了创建、删除、分页查询、归档等关键路径，可作为回归参考。

## 结论
ADK的数据库存储后端以清晰的接口分层与简洁的SQLite实现，提供了可靠的会话持久化能力。其软删除与归档机制兼顾了历史保留与性能优化，配合事务确保关键路径的一致性。对于生产部署，建议结合连接池、索引与监控策略进一步优化性能与稳定性，并在需要跨进程/跨节点场景下优先选择数据库后端。

### 数据库迁移与版本兼容
- **当前实现**：未包含自动迁移逻辑；建议在应用启动时执行Schema初始化脚本。
- **手动迁移**：可以通过执行初始化脚本或使用轻量迁移工具。
- **版本兼容性**：新增字段建议提供默认值；变更字段需考虑向后兼容与数据转换。

### 备份与恢复
- **SQLite文件备份**：SQLite文件即数据库；可通过复制数据库文件进行备份。
- **恢复策略**：恢复时注意停止写入，确保文件完整性；必要时启用WAL模式以降低锁竞争。
- **生产建议**：定期备份数据库文件，特别是在执行大量归档操作前后。

### 配置优化建议
- **连接池设置**：
  - MaxOpenConns: 25
  - MaxIdleConns: 25
  - ConnMaxLifetime: 5*time.Minute
  
- **SQLite参数优化**：
  - PRAGMA journal_mode=WAL;
  - PRAGMA synchronous=NORMAL;
  - PRAGMA cache_size=10000;
  
- **监控指标**：
  - 记录慢查询（>100ms）
  - 监控事务成功率
  - 跟踪归档操作耗时
